# 用户请求页面整体流程

本文描述用户访问「股票智能顾问」页面并完成一次分析请求的端到端流程。  
**当前架构**：A 在公网用 Nginx 处理请求，**前端在 B**；A 仅作代理（/api/proxy 供 B 访问外网）。

---

## 一、打开页面（首屏）

```
用户浏览器
    │  GET / 或 GET /index.html
    ▼
┌─────────────────────────────────────────────────────────┐
│ 服务器 A · Nginx (80/443)                                │
│ - location / → proxy_pass 到 B (8001)                     │
└─────────────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────────────┐
│ 服务器 B (端口 8001)                                      │
│ - FastAPI 静态：mount("/", frontend-react/dist)          │
│ - 返回 index.html + JS/CSS                               │
└─────────────────────────────────────────────────────────┘
    │  200 + 前端静态资源
    ▼
浏览器渲染：App.jsx → 表单页 (InputForm)
```

- **结论**：首屏经 **A 的 Nginx** 转发到 **B**，由 B 提供前端静态资源。

---

## 二、用户提交分析（核心流程）

用户填写公司名称、市场、仓位等，点击「开始分析」后：

```
1) 前端
   用户点击「开始分析」
   → 若未确认风险 → 弹出 RiskModal；确认后
   → startAnalysis(data)，切到 report 视图，挂载 ReportView

2) ReportView 发起 SSE
   EventSource("/api/analyze_sse?company_name=xxx&market=xxx&symbol=...&position=...")
   请求发往当前页面所在域名（经 A 的 Nginx）
```

```
3) 服务器 A · Nginx 收到 GET /api/analyze_sse?...
   location / 匹配，proxy_pass 到 B (8001)
```

```
4) 服务器 B 处理 /api/analyze_sse
   ├─ 解析公司名与标的
   │  resolve_company(company_name, market)
   │  → 优先 company_cache，未命中则 LLM 解析 → symbol、market
   │
   ├─ 获取行情（_fetch_market_data）
   │  fetch_single_price_with_fallback(market, symbol)
   │  ├─ A 股：Server B 本地 akshare
   │  └─ 港/美股：需访问外网（yfinance）
   │       ├─ 端口代理模式：B 设 HTTP_PROXY 指向 A 的 PROXY_PORT，yfinance 请求经 A 出网
   │       └─ API 代理模式：B 的 urlopen 被重定向到 A 的 POST /api/proxy/extern，A 请求 yfinance 相关 URL 后把结果回给 B
   │  → 得到 current_price、currency、as_of 等 → internal_market_data
   │
   ├─ 构建 Prompt
   │  build_user_prompt(company_name, market, extra_prompt, position, internal_market_data=...)
   │
   ├─ 调用 DeepSeek
   │  client.chat.completions.create(..., stream=True)
   │
   └─ SSE 流式输出
      先发 meta 事件：{ type: "meta", company_name, market, symbol, ... }
      再发正文内容（流式 chunk）
      最后发 data: [DONE]
```

```
5) 服务器 A · Nginx
   将 B 的 SSE 流按 chunk 转发给浏览器（proxy_buffering off）
```

```
6) 浏览器 ReportView
   EventSource onmessage：
   - 收到 meta → setMetaInfo，可写入历史
   - 收到正文 → setReportContent(prev => prev + text)，实时渲染 Markdown
   - 收到 [DONE] → 关闭连接，标记「分析完成」
```

## 三、流程简图（一次分析请求）

```
[用户浏览器]
     │ GET /
     ▼
[服务器 A · Nginx]  → proxy_pass 到 B
     │
     ▼
[服务器 B]  ← 静态资源（前端）
     │
     │ GET /api/analyze_sse?company_name=...&market=...
     ▼
[服务器 A · Nginx]  → proxy_pass 到 B（/ 与 /api/* 均到 B）
     │
     ▼
[服务器 B]
     │ ① resolve_company（缓存/LLM）
     │ ② fetch_single_price（A 股 akshare / 港美股 经 A 代理访问 yfinance）
     │ ③ build_user_prompt + internal_market_data
     │ ④ DeepSeek API 流式生成
     │ ⑤ SSE: meta → 正文流 → [DONE]
     │
     ▼
[服务器 A · Nginx]  ← 转发 SSE 流
     │
     ▼
[用户浏览器]  ← 实时展示报告、解析 meta、保存历史
```

---

## 四、涉及的外部依赖

| 环节           | 依赖说明 |
|----------------|----------|
| 服务器 A       | 公网 Nginx：/ 与 /api/* 转发到 B；本机 8000 仅提供 /api/proxy/extern 供 B 出网。 |
| 服务器 B       | 提供前端 + 业务 API；需 DeepSeek API Key；港/美股行情经 A 的 /api/proxy 访问外网。 |
| 前端           | 由 B 提供；分析请求 /api/analyze_sse 经 A 的 Nginx 转发到 B。 |

---

## 五、其他前端请求（简要）

- **健康检查**：`GET /health` → A 直接返回；若 Nginx 配置了 `/health` 代理到 A，则同样只经 A。
- **财务看板**：前端会请求 `/api/financials`、`/api/dcf`、`/api/pepb-band`。当前 Nginx 将 `/api/*` 全部转发到 B，若 B 实现这些接口即可使用。

以上即为用户请求页面并完成一次分析的整体流程。
